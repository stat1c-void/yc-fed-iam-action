# Agent Guidelines for yc-fed-iam-action

This document provides guidelines for automated coding agents working on this repository.

## Project Overview

This is a GitHub Action that retrieves Yandex Cloud IAM tokens through service account federation.
It's an ESM-based Node.js project targeting Node.js 20+.

---

## Build, Lint, and Test Commands

All commands should be run with the project's package manager (Yarn 4.x). Use `yarn` prefix for all
scripts.

### Standard Commands

- **Install dependencies**: `yarn install` (uses Yarn modern/berry)
- **Lint**: `yarn run lint` - Runs ESLint with strict rules
- **Format**: `yarn run format:write` - Auto-formats with Prettier
- **Format check**: `yarn run format:check` - Validates formatting without changes
- **Build/package**: `yarn run package` - Bundles with Rollup to `dist/action.js`
- **Test**: `yarn run test` - Runs Jest with coverage

### Running Single Tests

The project uses Jest with the testMatch pattern `**/*.test.js`. To run a specific test file:

```bash
yarn run test -- __tests__/main.test.js
```

To run a specific test by name:

```bash
yarn run test -- -t "works properly in happy path"
```

To run tests matching a pattern:

```bash
yarn run test -- __tests__/inputs
```

**Important**: The test script requires `NODE_OPTIONS=--experimental-vm-modules` (already
configured in package.json). Do not modify this unless Jest configuration changes.

## Code Style & Conventions

### Language & Standards

- **Module system**: ESM only (`"type": "module"` in package.json)
- **Node.js version**: >= 20 (specified in `engines` field)
- **Type checking**: JSDoc annotations with `@ts-check` directive
- **ECMAScript version**: ES2023 (configured in ESLint)

### File Structure

```text
src/           # Source files (ESM .js files)
  action.js    # Entry point for GitHub Action
  main.js      # Core business logic
__tests__/     # Jest test files (*.test.js)
dist/          # Build output (generated by Rollup)
coverage/      # Coverage reports (generated by Jest)
```

### Imports

- Use explicit imports; avoid namespace imports when possible
- Standard library imports: `import * as fs from "node:fs"` or
  `import { readFile } from "node:fs/promises"`
- Third-party imports: `import something from "package-name"` or
  `import { named } from "package-name"`
- Relative imports within project: include file extension `.js`

  ```javascript
  import main from "./main.js";
  ```

- Import order: Node standard library → third-party → local (enforced by eslint-plugin-import)

### Formatting

- **Line endings**: LF (`\n`) only
- **Indentation**: 2 spaces (no tabs)
- **Trailing commas**: ES5 style (only where valid)
- **Quote style**: Double quotes for strings
- **Arrow functions**: Always use parentheses around parameters, even for single param: `(x) => x`
- **Line length**: 99 characters
- **Line breaks**: Always add final newline

Prettier handles all formatting automatically. Run `yarn format:write` before committing.

### Type Safety & JSDoc

Use `// @ts-check` at the top of every `.js` file to enable TypeScript checking.

**JSDoc conventions**:

```javascript
/**
 * @typedef {object} TokenResponse
 * @property {string} token
 * @property {number} expiresIn
 */

/**
 * @param {string} idToken
 * @param {string} servAccId
 * @returns {Promise<TokenResponse>}
 */
async function getToken(idToken, servAccId) { ... }
```

- Use `@type`, `@param`, `@returns` for functions
- For objects, use `@typedef` to define structure
- Common types: `string`, `number`, `boolean`, `object`, `Promise<T>`, `{ key: value }`
- Prefer specific types over `any` or `unknown`

### Naming Conventions

- **Variables/functions**: `camelCase`
- **Constants**: `UPPER_SNAKE_CASE` only for true constants; otherwise `camelCase`
- **Files**: `kebab-case.js` for filenames
- **Classes** (if any): `PascalCase`
- **Test files**: `name.test.js` matching the source file being tested
- **Test describe blocks**: Use descriptive sentences or phrases
- **Test it blocks**: Start with verb in present tense: "returns token", "throws error"

### Error Handling

- **Top-level entry points**: Catch all errors and pass to `core.setFailed()`
- **Business logic functions**: Throw `Error` objects with descriptive messages
- **Network/API errors**: Include context (HTTP status, error codes) in error messages
- **External library errors**: Don't swallow; either handle meaningfully or re-throw
- **Validation errors**: Use `banditypes` for runtime validation; throw on invalid data
- **Console output**: Use `core.debug()`, `core.error()`, or `console.error()` appropriately

### Testing Conventions

- **Framework**: Jest 29+ with ESM support
- **File location**: Tests in `__tests__/` directory mirroring source structure
- **Test naming**: `*.test.js` files; imports use relative paths with `.js` extension
- **Test structure**:

  ```javascript
  import { describe, it, expect, beforeEach, afterEach, afterAll, jest } from "@jest/globals";

  describe("Feature name", () => {
    it("does something specific", async () => {
      // arrange, act, assert
    });
  });
  ```

- **Mocking**: Use `jest.unstable_mockModule()` for ESM module mocking; `jest.fn()` for function
  mocks
- **HTTP mocking**: Use `nock` with `nock.disableNetConnect()` and always `nock.restore()` in
  `afterAll`
- **Inputs**: Set via `process.env["INPUT_NAME"]` pattern
- **Assertions**: Use Jest matchers; prefer specific matchers like `toHaveBeenCalledWith`,
  `toHaveBeenNthCalledWith`
- **Coverage**: Enabled globally, outputs to `coverage/`; `.test.js` files auto-discovered

### GitHub Action Specifics

- Entry point: `src/action.js` → builds to `dist/action.js`
- `action.yml` defines inputs/outputs; must match `core.getInput()` keys
- Use `@actions/core` for all I/O (inputs, outputs, secrets, logging)
- Always call `core.setSecret()` for sensitive outputs
- Debug logging: `core.isDebug()` guard for verbose output
- Failures should call `core.setFailed()` with error message
